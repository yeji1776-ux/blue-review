# Coding Conventions

**Analysis Date:** 2026-04-06

## Naming Patterns

**Files:**
- React components: PascalCase (e.g., `LoginPage.jsx`, `BloggerMasterApp.jsx`)
- Hooks: camelCase with `use` prefix (e.g., `useAuth.js`)
- Utilities and libraries: camelCase (e.g., `supabase.js`)
- API handlers: kebab-case (e.g., `gcal-callback.js`, `gcal-token.js`)

**Functions:**
- React components: PascalCase with function declarations or const arrow functions (e.g., `function App()`, `const SortableTemplateItem = ({ t, onEdit }) => {}`)
- Event handlers: camelCase with `handle` prefix (e.g., `handleEmailChange`, `handleSubmit`, `handleBiometricDisable`)
- Helper functions: camelCase (e.g., `fetchWeather`, `requestLocation`, `saveProfile`, `copyToClipboard`)
- Callback/action functions: camelCase with `on` prefix (e.g., `onEdit`, `onSet`, `onUnlock`, `onSkip`)

**Variables:**
- State variables: camelCase (e.g., `authMode`, `email`, `password`, `rememberMe`)
- State setters: Auto-generated by `useState` with `set` prefix (e.g., `setAuthMode`, `setEmail`)
- Boolean flags: camelCase, often with `is`, `has`, or `can` prefix (e.g., `isRecovery`, `biometricAvailable`, `saveId`)
- Constants: UPPER_SNAKE_CASE for truly constant values (e.g., `EMAIL_DOMAINS`, `ADMIN_EMAILS`, `PLAN_LIMITS`)

**Types/Objects:**
- Component props: destructured as camelCase (e.g., `{ t, onEdit }`, `{ u, onSet }`)
- Objects: camelCase keys (e.g., `{ provider, redirectTo }`)

## Import Organization

**Order:**
1. React and React DOM (e.g., `import React, { useState, useEffect } from 'react'`)
2. Third-party libraries (e.g., `import { createClient } from '@supabase/supabase-js'`)
3. Icons/UI libraries (e.g., `import { Calendar, Clock, ... } from 'lucide-react'`)
4. Utilities and modules (e.g., `import { domToPng } from 'modern-screenshot'`)
5. Local hooks (e.g., `import { useAuth } from './hooks/useAuth'`)
6. Local components (e.g., `import LoginPage from './components/LoginPage'`)
7. Local utilities (e.g., `import { supabase } from './lib/supabase'`)
8. Drag-and-drop libraries (e.g., `import { DndContext, ... } from '@dnd-kit/core'`)

**Path Aliases:**
- No path aliases configured; uses relative imports (e.g., `../lib/supabase`, `./hooks/useAuth`)

## Error Handling

**Patterns:**
- Async functions return object destructuring: `{ data, error }` or `{ error }` (Supabase convention)
- Errors are checked with optional chaining: `result?.error`
- Error messages translated via helper functions: `translateError(msg)` in `src/components/LoginPage.jsx`
- Try-catch blocks used for error recovery:
  ```javascript
  try {
    // operation
  } catch (e) {
    // handle or log
  }
  ```
- Silent error handling with `.catch(() => {})` for non-critical operations
- User-facing error states stored in component state: `localError`, `authError`

**Error Translation:**
- `src/components/LoginPage.jsx` contains `translateError()` function that maps Supabase error messages to Korean user-friendly messages
- Example: `'Invalid login credentials'` → `'이메일 또는 비밀번호가 올바르지 않습니다.'`

## Logging

**Framework:** Console methods only; no logging library

**Patterns:**
- Basic console logging for errors: `console.error('이미지 저장 실패:', err)`
- Minimal logging in general; mostly error-focused
- Log messages in Korean when user-facing

## Comments

**When to Comment:**
- Comments are sparse in the codebase
- Used for section markers with dashes: `// ── 플랜 설정 ──` in `src/BloggerMasterApp.jsx`
- Inline comments for non-obvious logic (rare)
- Comments for Korean UI labels and explanations

**JSDoc/TSDoc:**
- Not used; no function documentation present

## Function Design

**Size:**
- Functions are generally small to medium
- Helper functions like `fetchWeather`, `saveProfile`, `copyToClipboard` are focused and single-purpose
- Event handlers are concise, often 1-5 lines

**Parameters:**
- Props destructured in function signature: `({ t, onEdit }) => {}`
- Async functions with clear intent: `signInWithEmail(email, password)`
- Handler functions receive React event objects directly: `(e) => {}`

**Return Values:**
- Functions return objects with `data` and `error` keys following Supabase convention
- Event handlers typically return void (used for side effects)
- Helper functions return values directly or null on error

## Module Design

**Exports:**
- Named exports: `export function useAuth() { ... }`
- Default exports: `export default App` for main components
- API handlers export default handler: `export default async function handler(req, res) { ... }`

**Barrel Files:**
- Not used; no index files aggregating exports

## Class vs Function Components

**Pattern:** Functional components with hooks exclusively

**Example:**
```javascript
function App() {
  return <BloggerMasterApp />
}

export default App
```

**Hooks Usage:**
- `useState` for local state
- `useEffect` for side effects and lifecycle
- `useRef` for DOM references and persistent values across renders
- `useMemo` for memoized computations
- Custom hooks: `useAuth()` for authentication state and methods

## Styling

**Approach:** Tailwind CSS with custom CSS classes

**Tailwind Usage:**
- Extensive use of utility classes: `className="min-h-screen bg-sky-50 flex items-center"`
- Custom theme defined in `src/index.css`:
  ```css
  --color-primary: #1d667e
  --color-on-surface: #0f172a
  --radius-card: 3.5rem
  ```

**Custom CSS Classes:**
- `.jelly-card` - Card with gradient background and glassmorphism
- `.jelly-button` - Interactive button with gradient and shadow effects
- `.ambient-blob` - Animated blurred background elements
- `.glass-card` - Glassmorphism effect with backdrop blur
- `.mesh-bg` - Mesh gradient background

**Colors:**
- Palette heavily uses sky colors (sky-50, sky-100, sky-200, etc.)
- Accent colors: amber, orange, rose, green for status/alerts
- Slate colors for text and neutral elements
